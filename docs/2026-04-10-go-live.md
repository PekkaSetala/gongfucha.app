**Why Caddy, not nginx:** one-line automatic HTTPS, no certbot cron, no renewal pain. Same resource footprint, dramatically less config. rag-spec.md:43 mentions nginx â€” that choice is superseded here.

**Why Next.js standalone output:** smallest container image, no `node_modules` copy, faster cold starts on a 2 GB VPS.

### Phase 2 â€” Containerization

**2.1 â€” `Dockerfile`** at repo root

Multi-stage, Next.js standalone, non-root user. Node 22 LTS **Debian slim** base (not alpine â€” see blocker below).

```dockerfile
# syntax=docker/dockerfile:1.7
FROM node:22-slim AS deps
WORKDIR /app
COPY package.json package-lock.json ./
RUN npm ci

FROM node:22-slim AS builder
WORKDIR /app
COPY --from=deps /app/node_modules ./node_modules
COPY . .
RUN npm run build

FROM node:22-slim AS runner
WORKDIR /app
ENV NODE_ENV=production
RUN groupadd --system app && useradd --system --gid app app
COPY --from=builder --chown=app:app /app/.next/standalone ./
COPY --from=builder --chown=app:app /app/.next/static ./.next/static
COPY --from=builder --chown=app:app /app/public ./public
USER app
EXPOSE 3000
CMD ["node", "server.js"]
```

**Why not alpine:** `src/lib/rag/embed.ts` imports `@huggingface/transformers`, which pulls `onnxruntime-node`. The prebuilt `onnxruntime-node` binaries are glibc-only; on Alpine (musl libc) the module crashes at `require` time, taking down the `/api/identify` route. `node:22-slim` is Debian-based (glibc), adds ~40 MB to the image, and avoids the problem entirely. Do not "optimize" back to alpine without replacing onnxruntime.

**Model cold-start caveat:** `@huggingface/transformers` downloads `onnx-community/all-MiniLM-L6-v2-ONNX` (~90 MB) from HuggingFace Hub on first call to `embedText`. Cached to the Transformers cache dir after that. First `/api/identify` call after every container restart is slow. Accepted for v1. Later optimization: pre-warm by copying the cache dir across build stages or calling the embedder during a post-build step.

**2.3 â€” `docker-compose.yml`** at repo root

```yaml
services:
  qdrant:
    image: qdrant/qdrant:latest
    restart: unless-stopped
    ports:
      - "127.0.0.1:6333:6333"   # localhost-only, for SSH-tunneled indexing from laptop
    volumes:
      - qdrant_data:/qdrant/storage
    networks:
      - internal

  app:
    build: .
    restart: unless-stopped
    environment:
      - NODE_ENV=production
      - NEXT_PUBLIC_SITE_URL=https://gongfucha.app
      - QDRANT_URL=http://qdrant:6333
      - OPENROUTER_API_KEY=${OPENROUTER_API_KEY}
    depends_on:
      - qdrant
    networks:
      - internal
      - web

  caddy:
    image: caddy:2-alpine
    restart: unless-stopped
    ports:
      - "80:80"
      - "443:443"
    volumes:
      - ./Caddyfile:/etc/caddy/Caddyfile:ro
      - caddy_data:/data
      - caddy_config:/config
    depends_on:
      - app
    networks:
      - web

networks:
  internal:
  web:

volumes:
  qdrant_data:
  caddy_data:
  caddy_config:
```

**Note:** `qdrant` and `app` share `internal`; only `caddy` and `app` are on `web`. Qdrant's `6333` port is bound to `127.0.0.1` on the VPS only â€” not reachable from the public internet, only via SSH tunnel. App â†’ Qdrant traffic goes over the internal docker network (`QDRANT_URL=http://qdrant:6333`), bypassing the port binding entirely. No API key needed.

### Phase 3 â€” VPS provisioning

Done manually in Hetzner Cloud console, not scripted (one-off).

1. **Create server:** CX22 (2 vCPU, 4 GB RAM, 40 GB disk), Ubuntu 24.04 LTS, Helsinki datacenter. Add SSH key at creation.
2. **First login:** `ssh root@<ip>`.
3. **Baseline hardening:**
   ```bash
   apt update && apt upgrade -y
   apt install -y ufw fail2ban curl
   curl -fsSL https://get.docker.com | sh     # official Docker repo; includes compose v2
   ufw allow 22/tcp
   ufw allow 80/tcp
   ufw allow 443/tcp
   ufw enable
   systemctl enable --now docker
   ```
   Do not use Ubuntu's `docker.io` / `docker-compose-plugin` packages â€” they lag and `docker-compose-plugin` isn't reliably available. The `get.docker.com` script installs from Docker's official apt repo.
4. **Create deploy user:**
   ```bash
   adduser --disabled-password --gecos "" deploy
   usermod -aG docker deploy
   mkdir -p /home/deploy/.ssh
   cp ~/.ssh/authorized_keys /home/deploy/.ssh/
   chown -R deploy:deploy /home/deploy/.ssh
   chmod 700 /home/deploy/.ssh
   chmod 600 /home/deploy/.ssh/authorized_keys
   ```
5. **Disable root SSH:** edit `/etc/ssh/sshd_config` â†’ `PermitRootLogin no`, then `systemctl restart ssh`. **Verify new session as `deploy` works before logging out.**
6. From now on, log in as `deploy@<ip>`.

## Risks and Rollback

| Risk                                                                | Mitigation                                                                                     |
|---------------------------------------------------------------------|------------------------------------------------------------------------------------------------|
| Transformers model download on first request = cold-start timeout  | Accepted for v1. Cold-start only affects the first `/api/identify` call after container boot. |
| Qdrant data volume lost on VPS rebuild                             | Acceptable â€” reindex from repo with `npm run rag:index` (takes ~30s for 84 entries).           |
| Let's Encrypt rate limit hit during testing                        | Caddy uses production endpoint by default; if you expect multiple test cycles, set `acme_ca https://acme-staging-v02.api.letsencrypt.org/directory` temporarily. |
| OPENROUTER_API_KEY committed by accident                           | `.env` in `.gitignore` (verify), `chmod 600`, pre-commit grep for `sk-or-`.                    |

**Rollback:** `docker compose down` â€” traffic drops, DNS still points at VPS. Fix forward, don't revert DNS.
